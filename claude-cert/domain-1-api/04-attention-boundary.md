# Domain 1.4 — 上下文容量 ≠ 模型注意力

> "Claude 有 200K context"不等于"你可以塞 200K 进去都能正常工作"。这一节讲的是**容量和注意力的区别**，以及怎么避免性能崩塌。

## 核心概念

**两个完全不同的概念**：

| 概念 | 英文 | 一句话 |
|------|------|-------|
| 上下文容量 | Context Window | 模型一次能"装下"的最大 token 数 |
| 注意力 | Attention | 模型对上下文内容的"理解和使用质量" |

**关键事实**：**容量是硬上限，注意力是软衰减。**

- 200K context = 200K token 能塞进去，不报错
- 但你塞到 100K 时，**注意力已经在衰减**：
  - 中间部分容易被"忽略"（lost in the middle）
  - 工具描述太多 → 模型选错工具
  - System prompt 被淹没 → 行为漂移

## 注意力衰减的几种表现

### 1. Lost in the middle（中间内容丢失）
长上下文里，**最开头和最结尾**的信息被关注最多。中间的容易被忽略。

例子：你在一个 80K 的对话中间塞了一条"用户偏好深色模式"，到第 100 轮，模型经常忘记。

### 2. Tool 描述过载
你定义了 50 个工具，每个工具描述 2KB → 总共 100KB 的工具 schema。
**结果**：
- 模型选工具的准确率下降
- 简单问题也要遍历所有工具描述
- Token 成本翻倍

### 3. 噪声覆盖信号
你把整个数据库 dump 进 context（"以防有用"）。
**结果**：真正重要的信息被噪声稀释，模型抓不到重点。

## 怎么避免这些问题

### 策略 1：分层管理上下文
- **System prompt**：稳定不变的角色、规则
- **Recent messages**：最近几轮对话原文
- **Summary**：更早的对话用结构化摘要替代
- **Retrieval**：需要时按需检索具体数据，不预先全塞

（这是 Domain 6 的核心内容，这里先建立直觉）

### 策略 2：工具子集化
不要一次给模型 50 个工具。
**正确做法**：
- 先做意图分类（routing）→ 只暴露相关的 5 个工具
- 或者实现"工具搜索"meta-tool，让模型按需发现工具
- MCP 的 `list_changed` 通知支持动态工具集

### 策略 3：压缩工具结果
工具返回 100KB JSON，但模型只需要其中 2 个字段。
**正确做法**：
- Tool 实现层就压缩，只返回模型需要的字段
- 或在 tool result 加工层提取关键信息

### 策略 4：分页 + cursor
返回搜索结果用 `{first_page: [...], cursor: "xxx", total: 1000}`，不要一次把 1000 条全塞回去。

## 容量和注意力的关系，类比一下

想象你在开会：
- **容量** = 会议室能塞 200 个人
- **注意力** = 你真正能听清谁在说话

塞 200 个人在会议室不会报错，但你听不清任何人。塞 10 个人，你听得很清楚。

## 考试常见陷阱

### 陷阱 1：以为"装得下"就是"用得好"
题目场景：「我们的应用 context 用了 150K，没超 200K 限制，但模型经常忘记早期信息。」
错误诊断："context 还没满，没问题。"
**正确诊断**：注意力已衰减，需要摘要压缩或检索方案。

### 陷阱 2：给模型几十个工具期待它选对
题目场景：「我接入了 30 个 MCP server 共 80 个工具，模型经常选错。」
错误做法：写更详细的工具描述（这进一步增加 token 噪声）。
**正确做法**：意图路由分流，或动态工具发现。

### 陷阱 3：以为 RAG 检索越多越好
检索 top-50 文档塞进 context → 模型抓不住重点。
**正确做法**：top-3 ~ top-5 高相关性结果 + 必要时让模型自己请求更多。

### 陷阱 4：把每个工具结果都完整保留
对话进行 20 轮后，每个工具结果都是 5KB JSON，全部保留 → context 爆炸 + 注意力分散。
**正确做法**：工具结果用完即压缩，只保留摘要 + 关键 ID。

## 用你的日常经验类比

你用 Claude Code 时，肯定遇到过：
- 进项目自动加载 `CLAUDE.md` + 最近 git log + 部分文件 → 这是**主动控制 context 信噪比**
- 长 session 后 Claude 开始遗忘最初的设定 → **注意力衰减**
- 你执行 `/clear` 重开 → 强制刷新 context

## 自检 3 题

**Q1**: 一个客服应用对话进行了 80 轮，每轮平均 3KB，总 context ~240KB。用户反馈"AI 忘事"。但 context 还没到 200K 上限（注：长 context 模型）。最可能的根因是？
- A. context 容量不够
- B. 注意力随长度衰减，中间内容丢失（lost in the middle）
- C. 模型版本太旧
- D. 网络延迟问题

<details>
<summary>答案</summary>
**B**。容量够≠注意力够。这是考"容量和注意力区别"的典型题。解决方案：摘要压缩 + 关键信息复述。
</details>

---

**Q2**: 一个 agent 接入了 60 个 MCP 工具，模型选错工具频率高达 30%。最优化方案是？
- A. 给每个工具描述加 "IMPORTANT: only use when..."
- B. 提高温度（temperature）让选择更稳定
- C. 实现意图路由，根据用户意图只暴露相关的 5-10 个工具
- D. 改用更大的模型

<details>
<summary>答案</summary>
**C**。工具过多 = 注意力分散。Routing/动态工具发现是标准方案。A 只会让 token 更臃肿，B 错（temp 不解决选择质量），D 治标不治本。
</details>

---

**Q3**: 一个 RAG 应用每次检索 top-30 文档塞进 context。用户反馈回答不准。最可能改进方向是？
- A. 检索 top-100 提高召回
- B. 减到 top-5 但提升检索精度，必要时让模型主动请求更多
- C. 用更小的模型
- D. 增加 system prompt 长度强调"请认真看"

<details>
<summary>答案</summary>
**B**。RAG 多 ≠ 好。少而精 + 按需检索是核心原则。A 加剧噪声，D 是 prompt 工程的伪解。
</details>

## 一句话总结

> **容量管"装得下"，注意力管"用得好"。优化方向：分层 context + 工具子集 + 结果压缩 + 按需检索。**
