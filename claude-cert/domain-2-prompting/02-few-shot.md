# Domain 2.2 — Few-shot 示例的威力

> 当你想让模型表达**微妙差别**时，**给 3 个例子 >> 写 500 字描述**。考试会反复测这个判断。

## 核心概念

**Few-shot prompting** = 在 prompt 里给模型几个"输入-输出"示例，让模型按示例的风格、格式、判断标准产生输出。

**核心威力**：
- 模型从例子里能学到**无法用语言完美描述**的东西
- 例如：「专业的语气」「保守的标注」「适度的简洁」

## 什么时候必须用 few-shot

### 场景 1：微妙判断标准
"这个用户评论是不是抱怨" —— 灰色地带很多。
- 干说"判断是否抱怨" → 模型把"建议"也标成抱怨
- 给 5 个边界例子 → 模型快速学会你的判断标准

### 场景 2：特定输出格式
"提取产品信息，包括隐含的尺寸" —— 抽象描述难传达。
- 干说"提取隐含信息" → 模型不知道要多激进
- 给 3 个例子展示什么算"隐含"、什么算"过度推断" → 一目了然

### 场景 3：边界情况
"什么情况下应该 escalate 给人工" —— 充满判断。
- 写 1000 字规则也不全
- 给 5 个真实场景（应/不应升级）→ 模型校准很准

### 场景 4：风格/语气
"用友好但专业的语气" —— 几乎无法用文字描述。
- 给 3 段示范文 → 模型完美模仿

## Few-shot 怎么写

### 推荐结构（用 XML）

```xml
<examples>
<example>
<input>用户评论："这个产品挺好用的，就是包装一般。"</input>
<output>
{
  "sentiment": "positive",
  "complaint": false,
  "issues": ["packaging"]
}
</output>
<reasoning>整体是积极的，"包装一般"是温和反馈，不是抱怨。</reasoning>
</example>

<example>
<input>用户评论："包装一般。"</input>
<output>
{
  "sentiment": "negative",
  "complaint": true,
  "issues": ["packaging"]
}
</output>
<reasoning>独立成句的负面反馈，没有积极对冲，算抱怨。</reasoning>
</example>
</examples>
```

### 关键设计要点

1. **覆盖边界**：包括"刚好该这样"和"刚好不该这样"的对比例子
2. **包含 reasoning**：让模型看到决策过程（不只是结果）
3. **多样性 > 数量**：3 个有代表性的例子 > 10 个相似例子
4. **错位例子**：故意放一个反例子说明"什么不算"

### 数量参考

| 任务类型 | Few-shot 数量 |
|---------|-------------|
| 简单分类 | 0-2 个 |
| 复杂判断 | 3-5 个 |
| 微妙边界 | 5-8 个 |
| 极复杂业务规则 | 8+ 个 + reasoning |

## 一个真实对比

**任务**：判断客服工单是否需要升级给主管。

### 用纯描述（无 few-shot）：
```
请判断是否需要 escalate。考虑：
- 用户情绪激烈
- 涉及金额大
- 反复投诉
- 提到法律/媒体
```
**结果**：模型误判率 ~25%。

### 用 5 个 few-shot：
```xml
<example>
<input>用户："这是我第三次反馈了，太离谱了"</input>
<output>{"escalate": true, "reason": "重复投诉"}</output>
</example>
<example>
<input>用户："我有点不满意，但我知道你们也尽力了"</input>
<output>{"escalate": false, "reason": "情绪缓和，理性反馈"}</output>
</example>
... (3 个边界例子)
```
**结果**：误判率 ~8%。

## 考试常见陷阱

### 陷阱 1：给重复的同类例子
5 个例子全是"明显抱怨"，没给"灰色地带"。
**结果**：模型在灰色地带还是不会判。
**正解**：每个例子覆盖一种**不同的情况**。

### 陷阱 2：例子的输出不带 reasoning
只给"输入 → 输出"，模型学到"对应关系"但不理解"为什么"。
**正解**：每个例子加 `<reasoning>` 字段，模型推理质量大幅提升。

### 陷阱 3：把 few-shot 当 prompt 的替代品
有些任务（如复杂业务规则）需要**规则+例子**结合。光给例子，模型可能泛化错方向。
**正解**：规则作为框架，例子作为校准。

### 陷阱 4：例子用真实敏感数据
把真实客户数据放例子里 → 隐私泄露 + prompt injection 风险。
**正解**：用脱敏/虚构数据。

### 陷阱 5：忘了 few-shot 的 token 成本
20 个长例子 = 几千 token，每次请求都发。
**正解**：精选 3-8 个高代表性例子。

## 用你的日常经验类比

你写 skill 的 SKILL.md 时，举的 `<example>` 块就是 few-shot。skill 之所以管用，很大程度是因为例子告诉了 Claude **何时**触发 skill、**长什么样**触发。

## 自检 3 题

**Q1**: 一个判断"用户评论是否包含产品功能建议"的任务，模型在明显场景判断对，在灰色场景错。最优改进方向？
- A. 把 system prompt 写得更详细
- B. 提高 temperature 让判断更稳定
- C. 加入 5-8 个覆盖**灰色边界**的 few-shot 例子，每个带 reasoning
- D. 切换更大的模型

<details>
<summary>答案</summary>
**C**。Few-shot 在灰色地带的校准能力，是单纯描述无法替代的。reasoning 让模型学到判断逻辑。
</details>

---

**Q2**: 一个 prompt 里有 15 个 few-shot 例子，全部是"高质量代码评审"的正面例子。模型在评审时仍然漏掉一些低质量代码。最可能根因是？
- A. 例子不够多，需要加到 30 个
- B. 缺少"应该指出问题但容易被忽略"的反例子，模型缺乏负向校准
- C. 模型版本太旧
- D. system prompt 没强调严格

<details>
<summary>答案</summary>
**B**。Few-shot 必须覆盖**对比例子**：正例 + 边界负例。15 个同质例子不如 5 个对比例子。
</details>

---

**Q3**: 关于 few-shot 例子的数量，以下哪个判断**正确**？
- A. 例子越多模型越准，越多越好
- B. 例子要尽可能相似，让模型学到一致模式
- C. 多样性比数量重要：5 个覆盖不同场景的例子 > 20 个相似例子
- D. 不要用 few-shot，纯描述更灵活

<details>
<summary>答案</summary>
**C**。Few-shot 核心是"代表性"和"边界覆盖"，不是数量。
</details>

## 一句话总结

> **Few-shot = 用例子说明无法用语言完美描述的标准。3-8 个覆盖**边界**的例子 + reasoning 字段，远胜千字描述。**
