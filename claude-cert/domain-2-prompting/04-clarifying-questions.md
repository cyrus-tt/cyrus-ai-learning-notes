# Domain 2.4 — 澄清问题与对话行为

> 模型遇到模糊请求，应该"问一句澄清"还是"按假设行动"？这是 production agent 设计的核心判断。考试很爱考。

## 核心概念

**模糊请求的处理有 4 种策略**：

| 策略 | 适用场景 | 风险 |
|------|---------|------|
| 直接执行（按最常见解释） | 低风险、易撤销 | 误解时浪费一次 |
| 显式假设 + 执行 | 中风险、需要透明 | 用户没看到假设就用了结果 |
| 提出一个聚焦问题 | 高风险、可逆性差 | 增加一次对话开销 |
| 列出多个澄清问题 | 极高风险、信息严重不足 | 用户烦躁 |

**核心原则**：
> **澄清问题的代价 = 多一次对话 vs 误解的代价 = 工作浪费/数据损坏。**
>
> **风险越高，越值得问一句。**

## 什么时候必须问

### 必须问的信号
1. **不可逆操作**：删除文件、发送邮件、扣款 → 必须确认
2. **多种合理解释**：用户说"清理一下"，是删旧文件还是格式化代码？解释差异大
3. **关键参数缺失**：日期范围、目标用户、阈值
4. **数据敏感性**：涉及 PII、生产数据库

### 可以直接做的信号
1. **可逆易撤销**：生成草稿、提取信息
2. **任务明确**：只有一种合理解释
3. **快速验证**：用户能立刻看到结果并纠正

## 提问的技巧

### 技巧 1：一次只问一个高价值问题
❌ 不好：
```
请问您想：
1. 用什么日期范围？
2. 包含哪些用户群？
3. 输出格式是 PDF 还是 Excel？
4. 是否需要图表？
5. 谁是接收人？
```
**问题**：用户一次回不全，对话冗长。

✅ 好：
```
开始之前先确认一个关键点：报告涵盖的日期范围是？
（其他细节我先按常见做法处理：用户群=全量，输出=Excel，含基础图表。如不符合请告诉我。）
```

### 技巧 2：明确假设，留撤回入口
当你判断"问题太多会烦用户"，又"不问可能出错"时：
```
我先按以下假设开始：
- 日期范围：最近 30 天
- 用户群：全量
- 输出：Excel
如果需要调整，告诉我。
```
**用户看到假设清单，能快速纠正**，比一开始什么都不说强很多。

### 技巧 3：高风险操作要预览 + 确认
不可逆操作（删除、发送、扣款）：
```
即将执行：删除 47 个文件（列表如下）
[文件列表]
确认删除？请回复 "确认" 或 "取消"。
```
**preview-token-execute** 模式：先 preview，得到一个一次性 token，下次调用时带 token 才真正执行。

## 不要做的事

### 1. 把"是否要问"硬编在 prompt 里
```
NEVER assume. ALWAYS ask before doing anything.
```
**结果**：模型变成"问题机器人"，连"帮我总结一下"都要先问 3 个问题。
**正解**：在 prompt 里写**原则**（"高风险问，低风险做"），配 few-shot 示例。

### 2. 高风险操作靠 prompt 控制
"删除前请确认"这种规则放 prompt → 容易被 prompt injection 绕过。
**正解**：tool 实现层强制 preview-confirm 两步流程。

### 3. 问完不记住假设
用户回了一次，下次对话又重新问。
**正解**：把用户的回答存到对话状态/structured state，后续对话引用。

### 4. 在每轮都重复确认
"我要删除文件 A 了，确认？" "好" "确认删除？" "好" "真的删除？" "我崩溃了"
**正解**：高风险**首次**确认，后续同类操作引用之前的确认。

## 一个真实场景

用户：「帮我清理一下这个项目」

### 不好的回应
```
请问您想：
1. 删除哪些类型的文件？
2. 是否包含 git 历史？
3. 是否需要保留某些目录？
4. 备份策略？
```
（用户回："哦算了，太麻烦")

### 好的回应
```
"清理项目" 我理解为以下三类，您选哪种？

1. **代码层面清理**：删未使用的 import、规范格式、整理无用变量。安全，可撤销。
2. **构建产物清理**：删 dist/、node_modules/、缓存等。可重新生成。
3. **完整重置**：包括 git untracked 文件、log 文件等。**不可逆**。

如果不确定，建议从 1 开始。
```

**这个回应**：
- 把模糊问题**分类**给用户选
- 标注**风险等级**
- 推荐**默认安全选项**

## 考试常见陷阱

### 陷阱 1：用"NEVER assume"强制模型问问题
模型变笨/啰嗦。原则化 + few-shot 才对。

### 陷阱 2：高风险操作只靠 prompt 提醒
**必须在 tool 实现里强制确认机制**。

### 陷阱 3：多个澄清问题一次问完
用户回答不全 → 对话扯不下去。
正解：一次一个最高价值问题。

### 陷阱 4：问得太多导致用户放弃
低风险任务直接做 + 显式假设 > 问 3 个问题。

### 陷阱 5：问完不存假设状态
导致后续对话反复问同样问题。

## 用你的日常经验类比

你用 Claude Code 时，Plan mode 就是"高风险任务先澄清"的体现 —— 大改动先 plan，小修复直接做。这个判断原则在所有 agent 设计里通用。

## 自检 3 题

**Q1**: 一个 customer service agent 接到"帮我处理一下这个订单"的模糊请求。最合理的做法？
- A. 直接按"取消订单"处理（最常见请求）
- B. 列 5 个澄清问题让用户选
- C. 提出一个聚焦问题："您是想取消、退款、还是修改地址？"，必要时列假设
- D. 拒绝处理直到用户说清楚

<details>
<summary>答案</summary>
**C**。聚焦问题 + 给选项是平衡效率和准确性的最佳做法。A 风险高（不可逆），B 烦用户，D 用户体验差。
</details>

---

**Q2**: 一个 agent 设计 prompt 时反复强调 "NEVER assume, ALWAYS ask"。实际使用中用户抱怨"啥都要问"。最优解？
- A. 改成 "SOMETIMES assume"
- B. 删除所有强制规则
- C. 改为原则化指导（高风险问、低风险做）+ few-shot 示例校准
- D. 切换更强模型

<details>
<summary>答案</summary>
**C**。强制规则导致行为僵化。原则化 + 例子是 prompt 工程标准做法。
</details>

---

**Q3**: 一个 agent 在 prompt 里写了"删除前必须确认"，但通过 prompt injection 用户可以绕过这个规则直接触发删除。根本解决方案？
- A. 改用更严格的措辞
- B. 加更多 IMPORTANT 标签
- C. 在 delete 工具实现层强制 preview-confirm 两步流程，prompt 只做引导
- D. 切换更安全的模型

<details>
<summary>答案</summary>
**C**。这考的是 prompt-vs-code 的核心原则：硬约束必须放代码。
</details>

## 一句话总结

> **澄清策略 = 风险权衡。高风险问一句、低风险直接做、中间地带显式假设。硬约束（确认机制）放代码不放 prompt。**
