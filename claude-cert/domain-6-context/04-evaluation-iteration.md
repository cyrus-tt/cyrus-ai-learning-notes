# Domain 6.4 — 评估、迭代与生产部署

> Production agent 的功夫一半在系统设计，一半在评估和迭代。这一节讲怎么让 agent **越用越好**而不是**越用越乱**。

## 核心概念

**反馈循环**：

```
观测（监控 + 评估）→ 诊断（系统性 vs 个例）→ 改进（schema / prompt / 工具）→ 重新评估 → ...
```

**关键原则**：
> **每次改动都要有可量化的"改进证据"，不靠"感觉"。**

## 评估的几个层次

### 1. 单元评估（Unit Eval）
针对单个函数 / 单个工具的固定测试集。

```
test_extract_amount:
  input: "Total: $1,234.56"
  expected_output: 1234.56
```

类似 unit test，每次改 prompt 跑一遍。

### 2. 集成评估（Integration Eval）
整个 pipeline 跑一组真实样本。

```
跑 100 张真实发票 → 比对人工标注 → 算准确率
```

### 3. 段位评估（Segment Eval）
**按维度细分**（Domain 6.3 讲过）。

```
按发票来源、按字段、按复杂度分别统计。
```

### 4. 线上评估（Production Eval）
真实流量上的指标：
- 错误率
- 用户反馈率
- Dismissal 率（人工复核被拒掉多少）
- Escalation 率

## 迭代的优先级

**改进顺序**（重要性递减）：

### 优先级 1：改 schema
最高 ROI。一个好 schema 比 1000 字 prompt 强。

### 优先级 2：改 prompt
- 加 few-shot 示例覆盖边界
- 改原则化引导
- 加 reasoning 字段

### 优先级 3：加工具
- 让模型能查更多信息
- 让模型能做更精确的操作

### 优先级 4：换模型
- 通常是最后一步
- 同代模型差距没你想象大

### 反优先级：堆基础设施
- 加缓存、加监控、加 retry 都很好
- 但**不能掩盖 prompt / schema 的根本问题**

## 系统性问题 vs 个例

判断错误是哪类，决定改什么：

| 信号 | 判断 |
|------|------|
| 100 条里有 5 条错，错的没规律 | 个例，可能是噪声 |
| 100 条里 30 条错，都集中在某字段 | 系统性，改 schema/prompt |
| 同一 prompt 跑 10 次结果不一致 | 不确定性，可能要降 temperature 或加结构 |
| 错的都是某类输入 | Coverage 问题，加 few-shot |

## 改进时的"对照实验"

每次改 prompt/schema：
1. 跑同一份评估集（baseline）
2. 改进版跑同一份
3. 比对准确率（按段）
4. 看是否有**回归**（某个段变差）

**典型坑**：
- 优化某个字段 → 整体平均提升
- 但**另一个字段悄悄退步**
- 不分段评估就发现不了

## Dismissal 是信号

如果你的 pipeline 有"人工复核拒掉"步骤：
- 拒掉的不是垃圾，是**模型出错的位置**
- 收集 dismissal 样本 → 这是最好的 few-shot 素材
- 系统性 dismissal → 改 schema/prompt

❌ **不要忽略 dismissal**。
✅ **把 dismissal 当成"产生的标注数据"**。

## 评估集的设计

### 1. 覆盖典型 + 边界 + 噩梦
- 80% 典型样本（验证常规质量）
- 15% 边界样本（验证健壮性）
- 5% 噩梦样本（破损 / 残缺 / 对抗）

### 2. 持续扩展
- 真实 production 错误 → 加入评估集
- 防止"上次修了，下次又坏了"

### 3. Stable + Mutating
- **Stable set**：永不变，作为长期基准
- **Mutating set**：定期更新，反映最新数据分布

## 生产监控的关键指标

### Tool-level
- 调用次数
- 失败率（按 error_type 分）
- 平均延迟
- Token 使用量

### Agent-level
- 任务完成率
- Escalation 率
- 重试率
- 用户满意度

### Cost-level
- 每个任务的平均 token 成本
- Batch vs 实时占比
- 成本异常告警

## 不要做的事

### 1. 改了不评估
"感觉好像变好了" → 实际可能某段崩了。

### 2. 整体准确率代替分段
看着 92% 没问题 → 关键字段 50%。

### 3. 把 dismissal 当噪声
那是免费标注数据。

### 4. 一次改多个变量
改了 schema + 改了 prompt + 换了模型 → 不知道哪个起作用。
**正解**：一次一个变量。

### 5. 用 production 数据当评估集
评估集应**永不进入训练 / 优化** → 避免泄漏。

### 6. 评估集太小
< 50 个样本 → 统计不显著。

## 一个完整的迭代示例

**场景**：发票提取，整体准确率 92%，但用户反馈"金额经常错"。

### Step 1：分段评估
分维度看：
- amount 字段准确率 **78%**  ← 找到了
- 其他字段都 > 95%

### Step 2：抽样错误
看 20 条 amount 错的案例：
- 12 条：千位逗号导致 OCR/模型错（1,234 → 1.234）
- 5 条：负数（退款）被当正数
- 3 条：多种货币混合

### Step 3：分类改进
- 千位逗号 → 加 few-shot 示例 + schema 注明格式
- 负数 → 加专门字段 `is_refund`，schema 允许负值
- 多货币 → 加 `currency` 字段强制识别

### Step 4：再评估
amount 准确率从 78% → 95%。
但 date 字段从 97% → 94%（轻微回归）。
**调查回归**：发现某个 few-shot 例子影响了 date 解析 → 修正 → date 回到 97%。

### Step 5：持续监控
线上跟踪 amount 错误率，加进 mutating eval set。

## 用你的日常经验类比

你做小红书内容 → 看数据（收藏/点赞比、留存）→ 改架构（加干货密度、改开头钩）→ 再发再看 → 形成爆款公式。这就是评估驱动的迭代。

## 自检 3 题

**Q1**: 一个 agent 整体准确率从 90% 改进到 93%。最关键的验证步骤？
- A. 庆祝胜利
- B. 部署上线
- C. 分段评估，确认没有任何字段或场景出现回归（变差）
- D. 跑更多样本

<details>
<summary>答案</summary>
**C**。整体上升可能掩盖局部下降，分段评估是发现回归的关键。
</details>

---

**Q2**: 人工复核拒掉了 8% 的 agent 输出。最合理处理？
- A. 视为噪声，忽略
- B. 这是免费的标注数据，收集错误案例改 schema/prompt，加进评估集
- C. 提高 confidence 阈值减少自动化
- D. 增加重试次数

<details>
<summary>答案</summary>
**B**。Dismissal 是迭代的金矿。系统性 dismissal 直接指向改进点。
</details>

---

**Q3**: 改进 agent 时，最推荐的优先级顺序？
- A. 先换更大模型 → 再改 prompt
- B. 先改 schema → 改 prompt → 加工具 → 最后才考虑换模型
- C. 先加缓存 + 监控 → 再改 prompt
- D. 同时改多个变量，效率高

<details>
<summary>答案</summary>
**B**。改进 ROI 顺序：schema > prompt > 工具 > 模型。一次一个变量。基础设施重要但不能掩盖根本问题。
</details>

## 一句话总结

> **评估驱动迭代：分段评估 + 对照实验 + Dismissal 当数据。改进优先级：schema > prompt > 工具 > 模型。一次一个变量。Production 监控要细到每个工具和字段。**
